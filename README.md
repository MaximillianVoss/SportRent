# SportRent

Курсовой проект на тему: **«Разработка мобильного приложения для аренды спортивного инвентаря»**.

Проект представляет собой мобильное приложение на **.NET MAUI** с локальной базой данных **SQLite**. Приложение предназначено для просмотра каталога спортивного инвентаря, авторизации пользователя, оформления аренды, просмотра заказов и личного кабинета.

## Назначение проекта

Цель проекта:

- спроектировать предметную область аренды спортивного инвентаря;
- разработать структуру базы данных для хранения пользователей, инвентаря, тарифов, пунктов проката, заказов и платежей;
- реализовать мобильное клиентское приложение с локальной БД;
- обеспечить демонстрационный сценарий для курсовой работы без зависимости от внешнего сервера.

Проект ориентирован на учебную демонстрацию полного цикла:

1. проектирование базы данных;
2. генерация SQLite-базы из SQL-скриптов;
3. scaffold сущностей EF Core;
4. разработка мобильного интерфейса;
5. демонстрация пользовательских сценариев.

## Состав решения

Решение включает три основных проекта.

### `SportRent.Mobile`

Мобильное приложение на .NET MAUI.

Реализовано:

- экран входа с демо-аккаунтами;
- каталог инвентаря;
- карточка инвентаря;
- оформление аренды;
- просмотр заказов;
- личный кабинет пользователя;
- локальная работа с SQLite без подключения к серверу.

### `SportRent.DbTool`

Консольный инструмент для сборки базы данных SQLite из SQL-скриптов.

Функции:

- поиск нумерованных SQL-скриптов в папке `database`;
- последовательное выполнение скриптов;
- создание итогового файла базы данных;
- безопасная замена существующей БД через временный файл;
- возможность указать корень проекта, директорию скриптов и путь вывода.

### `SportRent.Data`

Слой данных, содержащий:

- scaffolded-сущности EF Core;
- `SportRentDbContext`;
- PowerShell-скрипт для повторной генерации сущностей из SQLite.

## Архитектура решения

### Общая схема

Проект разделен на слои:

- SQL-слой: схема и начальные данные в `database`;
- инструментальный слой: `SportRent.DbTool` собирает файл `sportRent.db`;
- data-layer: `SportRent.Data` содержит сущности и `DbContext`;
- presentation-layer: `SportRent.Mobile` содержит интерфейс, ViewModel и сервисы доступа к локальной БД.

### Архитектура мобильного приложения

В мобильном приложении используется упрощённый подход MVVM:

- `Pages` отвечают за визуальный интерфейс;
- `ViewModels` содержат состояние экрана и команды;
- `Services` работают с локальной SQLite;
- `Models` описывают DTO и представления данных для UI.

Приложение работает с **копией базы данных**, которая:

1. встраивается в ресурсы приложения;
2. при первом запуске копируется в `AppDataDirectory`;
3. используется как рабочая локальная БД;
4. может пересоздаваться при изменении версии встроенной базы данных.

Это позволяет:

- хранить пользовательские заказы локально;
- не терять изменения в ходе одного жизненного цикла установленного приложения;
- обновлять локальную БД при выпуске новой версии приложения.

### Основные сущности предметной области

Справочники:

- роли пользователей;
- статусы заказов;
- методы оплаты;
- статусы платежей;
- состояния инвентаря;
- типы аренды;
- категории;
- бренды;
- типы инвентаря;
- размеры.

Операционные сущности:

- пользователи;
- пункты проката;
- инвентарь;
- тарифы аренды;
- остатки инвентаря по пунктам проката;
- изображения инвентаря;
- заказы аренды;
- позиции заказа;
- платежи.

### Ключевые особенности схемы

- деньги хранятся как `INTEGER` в копейках;
- тарифы вынесены в отдельную таблицу `equipmentRates`;
- остатки инвентаря хранятся по точкам проката в `rentalPointEquipment`;
- изображения нормализованы через `images` и `equipmentPhotos`;
- заказ связан с пользователем, статусом, пунктом выдачи и пунктом возврата;
- позиции заказа ссылаются на конкретный остаток `rentalPointEquipment`.

## Реализованные пользовательские сценарии

### 1. Авторизация

В приложении есть экран входа с демо-аккаунтами.

Предзаполненный вход:

- email: `ivan.petrov@sport-rent.local`
- пароль: `client123`

Также доступны демо-аккаунты менеджера и администратора, загружаемые из таблицы `users`.

### 2. Просмотр каталога

Пользователь может:

- просматривать весь список инвентаря;
- искать по названию, бренду, типу и категории;
- фильтровать позиции по категориям;
- видеть стартовую цену, залог и доступность по пунктам проката.

### 3. Просмотр карточки инвентаря

В карточке инвентаря отображаются:

- название и категория;
- бренд и модель;
- описание;
- локальное изображение;
- тарифы аренды;
- доступность по пунктам проката;
- кнопка перехода к оформлению аренды.

### 4. Оформление аренды

Пользователь может:

- выбрать тариф;
- выбрать пункт выдачи;
- указать дату и время начала аренды;
- выбрать количество единиц;
- указать количество периодов аренды;
- получить расчет стоимости аренды, залога и общей суммы;
- создать заказ.

При оформлении:

- уменьшается `availableQuantity` в `rentalPointEquipment`;
- создается запись в `rentOrders`;
- создается запись в `orderItems`;
- создается запись в `payments` со статусом ожидания оплаты.

### 5. Просмотр заказов

В разделе заказов пользователь видит:

- историю заказов;
- статус заказа;
- сроки аренды;
- пункт выдачи и возврата;
- сумму аренды;
- сумму залога;
- сумму платежа;
- статус оплаты.

### 6. Личный кабинет

В профиле отображаются:

- ФИО пользователя;
- роль;
- email;
- телефон;
- дата регистрации;
- количество заказов;
- количество активных и завершенных заказов;
- общая сумма оплаченных заказов;
- сумма ожидающих оплат;
- кнопка выхода из аккаунта.

## Изображения инвентаря

Для демонстрационного режима изображения инвентаря хранятся как локальные ресурсы MAUI в папке:

- `SportRent.Mobile/Resources/Images`

В таблице `images.url` сохраняется имя локального asset-файла, например:

- `equipment_bike.jpg`
- `equipment_skis.jpg`
- `equipment_snowboard.jpg`
- `equipment_scooter.jpg`
- `equipment_helmet.jpg`

Такой подход выбран специально для курсовой работы, чтобы:

- не зависеть от внешних URL;
- избежать проблем с недоступностью сети;
- гарантировать отображение фото на защите проекта.

## Сборка базы данных

### DbTool

Сборка SQLite выполняется через `SportRent.DbTool`.

Пример запуска из корня решения:

```powershell
dotnet run --project SportRent.DbTool -- --root . --scripts-dir database --output SportRent.Mobile\Resources\Raw\Database\sportRent.db
```

Что делает инструмент:

1. ищет SQL-скрипты формата `000_name.sql`;
2. сортирует их по имени;
3. создаёт временную SQLite БД;
4. выполняет скрипты в транзакции;
5. заменяет итоговый файл БД только после успешного выполнения.

### Scaffold слоя данных

Для пересоздания `DbContext` и сущностей EF Core используется:

```powershell
.\SportRent.Data\scaffold.ps1
```

Скрипт:

- пересобирает SQLite;
- очищает старые scaffolded файлы;
- запускает `dotnet ef dbcontext scaffold`.

## Запуск проекта

### Требования

- Windows 10/11 для проверки Windows-таргета;
- .NET SDK 9;
- установленный workload MAUI;
- PowerShell 5+ или PowerShell 7+;
- доступ к NuGet для первичного восстановления пакетов.

### Сборка мобильного приложения

Проверенная команда сборки:

```powershell
dotnet build SportRent.Mobile\SportRent.Mobile.csproj -f net9.0-windows10.0.19041.0
```

### Рекомендуемая последовательность перед запуском

```powershell
dotnet run --project SportRent.DbTool -- --root . --scripts-dir database --output SportRent.Mobile\Resources\Raw\Database\sportRent.db
dotnet build SportRent.Mobile\SportRent.Mobile.csproj -f net9.0-windows10.0.19041.0
```

### Первый запуск приложения

При первом запуске:

1. встроенная база данных копируется в локальную рабочую директорию приложения;
2. открывается экран входа;
3. уже подставлены данные клиента;
4. после входа становится доступен каталог и дальнейшие пользовательские сценарии.

Если встроенная база данных была обновлена, приложение пересоздаст рабочую копию на старте автоматически.

## Текущие ограничения

Проект является учебным и не претендует на production-ready уровень.

Ограничения текущей версии:

- авторизация демонстрационная, пароль хранится в seed-данных без хеширования;
- отсутствует серверная часть и синхронизация с удаленной БД;
- отсутствует регистрация новых пользователей;
- отсутствует реальная интеграция с платежной системой;
- нет полноценной ролевой логики интерфейса для менеджера и администратора;
- нет автоматизированных тестов;
- основной сценарий проверки на данный момент ориентирован на Windows-сборку MAUI.
